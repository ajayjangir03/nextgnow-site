5G Registration Call Flow

Updated: Apr 20, 2026 Live · 3GPP Spec 📖 8 min read

Overview

The 5G Registration procedure is the fundamental call flow by which a UE attaches to the 5G Core network (5GC) and establishes its initial registration state with the Access and Mobility Management Function (AMF). This procedure enables the UE to obtain network services, receive a 5G-GUTI (Globally Unique Temporary Identity), and establish security context through mutual authentication with the network. The registration type can be Initial Registration, Mobility Registration Update, or Periodic Registration Update depending on the UE state and trigger.

Registration occurs over the N1 interface (NAS signalling) between UE and AMF, with the gNB providing transparent relay of NAS messages over the N2 interface. The AMF orchestrates authentication via AUSF, retrieves subscription data from UDM, and may invoke PCF for policy and SMF for PDU session establishment if requested by the UE during registration.

  • Establish UE identity in 5GC using SUCI or 5G-GUTI
  • Perform mutual authentication and establish NAS security context
  • Register UE mobility state and assign temporary identities
  • Enable network slicing selection and QoS policy enforcement
  • Optionally establish PDU sessions for data connectivity

Network Functions Involved

UE User Equipment initiating registration to 5GC
gNB Next Generation NodeB providing RAN access and NAS relay
AMF Access and Mobility Management Function orchestrating registration
AUSF Authentication Server Function performing 5G AKA or EAP-AKA'
UDM Unified Data Management storing subscription data and authentication vectors
UDR Unified Data Repository providing persistent storage for subscriber information
PCF Policy Control Function providing access and mobility policies
SMF Session Management Function establishing PDU sessions if requested

System Architecture

UE ⇄ gNB ⇄ AMF ⇄ AUSF ⇄ UDM/UDR, AMF ⇄ PCF, AMF ⇄ SMF
5G Core Network Reference Architecture (3GPP TS 23.501, TS 23.502)

Procedure Steps

1
Registration Request
UE → AMF N1
The UE sends a Registration Request message to the AMF via gNB containing its identity (either SUCI for privacy-protected SUPI or 5G-GUTI from previous registration), the registration type (Initial, Mobility, Periodic), requested network slices (S-NSSAI), and security capabilities. The message also includes mobility context such as the last visited TAI and may include a PDU Session Establishment Request if the UE requires immediate data connectivity.
Registration TypeSUCI or 5G-GUTIRequested NSSAIUE Security CapabilityLast Visited TAI
2
UE Identity Retrieval (if SUCI provided)
AMF → UDM Nudm
If the UE provided a SUCI (Subscription Concealed Identifier) instead of 5G-GUTI, the AMF invokes the UDM via Nudm_UECM_Get service operation to de-conceal the SUCI and retrieve the permanent SUPI (Subscription Permanent Identifier). The UDM uses the home network private key corresponding to the public key used by the UE to decrypt the SUCI and returns the SUPI to the AMF for subscriber identification.
SUCIHome Network Public Key
3
Authentication Initiation Request
AMF → AUSF Nausf
The AMF invokes the AUSF via Nausf_UEAuthentication_Authenticate service operation, providing the SUPI, serving network name, and ABBA (Anti-Bidding down Between Architectures) parameters. The AUSF initiates the authentication procedure, selecting either 5G AKA (Authentication and Key Agreement) or EAP-AKA' based on the subscription profile and home network policy.
SUPIServing Network NameABBA parameters
4
Authentication Vector Request
AUSF → UDM Nudm
The AUSF requests authentication vectors from the UDM via Nudm_UEAuthentication_Get service operation, providing the SUPI and serving network name. The UDM retrieves the subscriber's long-term key K and generates authentication vectors (5G AV) containing RAND, AUTN, XRES*, and KAUSF, then returns these to the AUSF for use in the authentication challenge.
SUPIServing Network NameAUSF Instance ID
5
Authentication Request
AMF → UE N1
The AMF sends an Authentication Request message to the UE containing the random challenge (RAND), authentication token (AUTN), key set identifier (ngKSI), and ABBA parameters received from AUSF. The UE verifies the AUTN to authenticate the network and, if successful, computes the response RES using its USIM and the received RAND.
RANDAUTNngKSIABBA
6
Authentication Response
UE → AMF N1
The UE sends an Authentication Response message containing RES* (the hashed authentication response) to the AMF. The UE has verified the network authenticity via AUTN and computed RES* from RES using the same key derivation as the network. The UE also derives the anchor key KAUSF and subsequently KSEAF and KAMF for NAS security.
RES*
7
Authentication Confirmation
AMF → AUSF Nausf
The AMF forwards the received RES* to the AUSF for verification. The AUSF compares the received RES* with the expected XRES* (derived from the authentication vector). If they match, authentication is successful and the AUSF confirms authentication success to the AMF, providing KSEAF (Security Anchor Function key) for establishing NAS security context.
RES*
8
Subscription Data Retrieval
AMF → UDM Nudm
The AMF invokes Nudm_SDM_Get service operation to retrieve subscription data from UDM including access and mobility subscription data, subscribed S-NSSAIs (network slices), UE context in AMF, and SMF selection subscription data. The UDM retrieves this information from UDR and returns it to AMF for registration authorization and policy enforcement.
SUPIPLMN IDRequested Data Sets
9
UE Context Registration
AMF → UDM Nudm
The AMF registers itself as the serving AMF for the UE by invoking Nudm_UECM_Registration with the SUPI, AMF instance ID, GUAMI (Globally Unique AMF Identifier), and RAT type. The UDM stores this context and may trigger deregistration at any previous AMF serving the UE. This enables UDM to route mobile-terminated services to the correct AMF.
SUPIAMF Instance IDGUAMIRAT Type
10
Policy Association Establishment
AMF → PCF Npcf
The AMF establishes a policy association with PCF via Npcf_AMPolicyControl_Create service operation, providing UE identity, access type, and selected network slice information. The PCF returns Access and Mobility Policy including allowed/restricted areas, UE Route Selection Policy, and RFSP index. These policies govern UE mobility behavior and service access rights.
SUPIDNNS-NSSAIAccess Type
11
NAS Security Mode Command
AMF → UE N1
The AMF initiates NAS security by sending a Security Mode Command to the UE, selecting NAS integrity and ciphering algorithms from the intersection of network capabilities and UE security capabilities. The message includes the key set identifier (ngKSI) and replays the UE security capabilities to prevent bidding-down attacks. This message is integrity protected using KAMF-derived keys.
Selected NAS Security AlgorithmsngKSIReplayed UE Security Capabilities
12
NAS Security Mode Complete
UE → AMF N1
The UE verifies the Security Mode Command (including replay of security capabilities) and if successful, activates the selected NAS security algorithms. The UE sends Security Mode Complete message, which is both integrity protected and ciphered. The message may include the IMEISV (International Mobile Equipment Identity and Software Version) if requested by the network for equipment identification.
IMEISV (optional)NAS Message Container
13
Initial Context Setup Request
AMF → gNB N2
The AMF sends an Initial Context Setup Request to the gNB over N2 interface (NGAP protocol) to establish the UE context in RAN. The message includes security keys for AS (Access Stratum) security, allowed network slices, UE aggregate maximum bit rate, mobility restrictions, and PDU session resource setup list if PDU sessions are being established concurrently with registration.
AMF UE NGAP IDAllowed NSSAIUE Security CapabilitiesSecurity KeyMobility Restrictions
14
AS Security Mode Command
gNB → UE RRC
The gNB initiates Access Stratum security by sending RRC Security Mode Command to the UE, selecting AS integrity and ciphering algorithms. The UE verifies the algorithms and activates AS security. All subsequent RRC signalling and user plane data will be protected using these algorithms with keys derived from KAMF. The NCC (Next hop Chaining Counter) enables key refresh during handovers.
Selected AS Security AlgorithmsNCC
15
AS Security Mode Complete
UE → gNB RRC
The UE confirms activation of AS security by sending RRC Security Mode Complete message to the gNB. This message is integrity protected and ciphered using the activated AS security context. The gNB verifies successful security activation and proceeds with registration context establishment.
Selected UE Security Capabilities
16
Initial Context Setup Response
gNB → AMF N2
The gNB confirms successful establishment of UE context by sending Initial Context Setup Response to AMF. The message includes RAN UE NGAP ID for future N2 signalling, and if PDU sessions were requested, includes PDU Session Resource Setup Response List with GTP-U tunnel information for user plane. The UE is now registered at both RAN and core network levels.
AMF UE NGAP IDRAN UE NGAP IDPDU Session Resource Setup Response List
17
Registration Accept
AMF → UE N1
The AMF sends a Registration Accept message to the UE confirming successful registration. The message includes the assigned 5G-GUTI (temporary identity for subsequent procedures), allowed network slices (S-NSSAIs), Tracking Area Identity list defining the registration area, periodic registration update timer, and optionally mobility restrictions and emergency service support indicators. This message is NAS security protected.
5G-GUTIAllowed NSSAITAI ListRegistration AreaPeriodic Update Timer
18
Registration Complete
UE → AMF N1
The UE acknowledges the Registration Accept by sending a Registration Complete message to the AMF. The UE stores the assigned 5G-GUTI, allowed NSSAIs, registration area, and other parameters. The message may include SORT (SMS over NAS transport) if SMS service is being activated. The UE is now in 5GMM-REGISTERED state and can access network services including establishment of PDU sessions for data connectivity.
SORT (optional)
